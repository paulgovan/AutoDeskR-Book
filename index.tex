\chapter*{Preface}\label{preface}
\addcontentsline{toc}{chapter}{Preface}

\markboth{Preface}{Preface}

\emph{R and the AutoDesk Platform} documents the
\href{https://github.com/paulgovan/AutoDeskR}{AutoDeskR} R package,
which provides an R interface to the
\href{https://aps.autodesk.com}{AutoDesk Platform Services (APS)} APIs.
APS is a cloud platform that exposes 3D model processing, file storage,
and interactive viewing capabilities through a REST API. This book shows
how to access those capabilities directly from R, enabling data
scientists and engineers to incorporate CAD and BIM workflows into their
existing R-based analyses, reports, and Shiny applications.

\section*{What You Will Learn}\label{what-you-will-learn}
\addcontentsline{toc}{section}{What You Will Learn}

\markright{What You Will Learn}

The book is organized into chapters covering each major API:

\begin{itemize}
\tightlist
\item
  \textbf{Authentication} --- OAuth token setup using
  \texttt{getToken()} and managing API credentials securely
\item
  \textbf{Data Management} --- Bucket creation, file upload, and object
  management in AutoDesk's Object Storage Service (OSS)
\item
  \textbf{Model Derivative} --- Translating design files into OBJ, STL,
  and SVF formats for downstream analysis
\item
  \textbf{Design Automation} --- Automated cloud-based DWG-to-PDF
  conversion without a local AutoCAD installation
\item
  \textbf{Viewer} --- Embedding the AutoDesk WebGL viewer in R Markdown
  documents and Shiny applications
\item
  \textbf{Troubleshooting} --- Solutions to common issues encountered
  when working with the APS APIs
\end{itemize}

\section*{The Problem}\label{the-problem}
\addcontentsline{toc}{section}{The Problem}

\markright{The Problem}

AutoDesk Platform Services (APS) is one of the most capable cloud
platforms for working with CAD and BIM data. It can translate design
files between dozens of formats, extract rich geometry and metadata from
models, automate DWG processing at scale, and render 3D models
interactively in a browser --- all without requiring a local AutoCAD or
Revit installation.

The challenge is that APS speaks REST. Every operation --- uploading a
file, kicking off a translation job, querying model metadata ---
requires constructing authenticated HTTP requests, handling
base64-encoded URNs, polling asynchronous job queues, and parsing nested
JSON responses. For engineers and architects who live in R, this
friction is enough to make APS feel out of reach.

\section*{The Opportunity}\label{the-opportunity}
\addcontentsline{toc}{section}{The Opportunity}

\markright{The Opportunity}

R has become the lingua franca for quantitative work in architecture,
engineering, and construction (AEC) analytics. Structural engineers use
it to process sensor data. Sustainability consultants use it to model
energy performance. BIM managers use it to audit model quality across
large portfolios. Shiny brings these analyses to non-technical
stakeholders through interactive web applications.

What has been missing is a direct connection between R's data science
ecosystem and the 3D model data locked inside CAD files. A DWG file is
not just a drawing --- it contains geometry, layer structure, material
properties, and embedded attributes that are genuinely useful for
analysis. An SVF model rendered in a browser can be linked to live data
in a Shiny dashboard. A PDF exported from a DWG can be generated
automatically as part of a reporting pipeline.

\section*{The Solution}\label{the-solution}
\addcontentsline{toc}{section}{The Solution}

\markright{The Solution}

AutoDeskR wraps the APS REST API in idiomatic R functions, handling
authentication, encoding, and job polling so that you don't have to. A
workflow that would otherwise require dozens of raw \texttt{httr} calls
and careful reading of API documentation is reduced to a few lines:

\begin{Shaded}
\begin{Highlighting}[]
\FunctionTok{library}\NormalTok{(AutoDeskR)}

\NormalTok{token  }\OtherTok{\textless{}{-}} \FunctionTok{getToken}\NormalTok{(id, secret, }\AttributeTok{scope =} \StringTok{"data:read data:write bucket:create"}\NormalTok{)}
\NormalTok{bucket }\OtherTok{\textless{}{-}} \FunctionTok{makeBucket}\NormalTok{(token, }\StringTok{"my{-}project{-}bucket"}\NormalTok{)}
\NormalTok{upload }\OtherTok{\textless{}{-}} \FunctionTok{uploadFile}\NormalTok{(token, }\StringTok{"model.dwg"}\NormalTok{, bucket}\SpecialCharTok{$}\NormalTok{bucketKey)}
\NormalTok{job    }\OtherTok{\textless{}{-}} \FunctionTok{translateFile}\NormalTok{(token, upload}\SpecialCharTok{$}\NormalTok{objectId, }\StringTok{"svf"}\NormalTok{)}
\end{Highlighting}
\end{Shaded}

The goal is not to abstract APS away entirely --- understanding the
underlying API makes you more effective. Rather, AutoDeskR removes the
boilerplate so you can spend your time on the analysis, not the
plumbing.

\section*{Step 2: Register an APS
Application}\label{step-2-register-an-aps-application}
\addcontentsline{toc}{section}{Step 2: Register an APS Application}

\markright{Step 2: Register an APS Application}

APS uses OAuth 2.0. Every API call must be authenticated with a
\textbf{Client ID} and \textbf{Client Secret} that belong to a
registered application.

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  Go to the \href{https://aps.autodesk.com}{APS Developer Portal} and
  sign in.
\item
  Click \textbf{My Apps → Create Application}.
\item
  Give your app a name (e.g., \texttt{AutoDeskR-dev}) and select the
  APIs you plan to use. For the examples in this book, enable
  \textbf{Data Management}, \textbf{Model Derivative}, and
  \textbf{Design Automation}.
\item
  Click \textbf{Create}. Your \textbf{Client ID} and \textbf{Client
  Secret} will be displayed on the app's detail page.
\end{enumerate}

Keep these credentials private. Never commit them to version control.

\section*{Step 3: Store Credentials
Securely}\label{step-3-store-credentials-securely}
\addcontentsline{toc}{section}{Step 3: Store Credentials Securely}

\markright{Step 3: Store Credentials Securely}

The safest way to use credentials in R is through environment variables
stored in \texttt{\textasciitilde{}/.Renviron}. Open or create that
file:

\begin{Shaded}
\begin{Highlighting}[]
\NormalTok{usethis}\SpecialCharTok{::}\FunctionTok{edit\_r\_environ}\NormalTok{()}
\end{Highlighting}
\end{Shaded}

Add the following lines, replacing the placeholder values:

\begin{verbatim}
APS_CLIENT_ID=your_client_id_here
APS_CLIENT_SECRET=your_client_secret_here
\end{verbatim}

Save the file and restart R. You can then retrieve the values in any
script without hard-coding them:

\begin{Shaded}
\begin{Highlighting}[]
\NormalTok{id     }\OtherTok{\textless{}{-}} \FunctionTok{Sys.getenv}\NormalTok{(}\StringTok{"APS\_CLIENT\_ID"}\NormalTok{)}
\NormalTok{secret }\OtherTok{\textless{}{-}} \FunctionTok{Sys.getenv}\NormalTok{(}\StringTok{"APS\_CLIENT\_SECRET"}\NormalTok{)}
\end{Highlighting}
\end{Shaded}

See the \href{authentication.qmd}{Authentication} chapter for a full
explanation of OAuth scopes and token management.

\section{Store Credentials Safely}\label{store-credentials-safely}

Store your Client ID, Secret, and access tokens in
\texttt{\textasciitilde{}/.Renviron} and access them with
\texttt{Sys.getenv()} to keep credentials out of your code and version
control.

entries:

\begin{verbatim}
client_id = "YOUR_CLIENT_ID"
client_secret = "YOUR_CLIENT_SECRET"
token = "YOUR_ACCESS_TOKEN"
urn = "YOUR_URN"
\end{verbatim}

\chapter{Reality Capture}\label{reality-capture}

\chapter{Reality Capture}\label{reality-capture-1}

The Reality Capture API (also called Photo-to-3D) converts a set of
overlapping photographs into a 3D mesh or point cloud. You upload images
--- from a phone, DSLR, or drone --- and AutoDesk's cloud photogrammetry
engine stitches them into a georeferenced 3D model you can download or
feed into the viewer.

Common use cases include:

\begin{itemize}
\tightlist
\item
  \textbf{As-built documentation} --- capture a construction site weekly
  with a drone and generate a mesh for each visit
\item
  \textbf{Heritage and inspection} --- record a structure or asset in 3D
  from photos taken by field staff
\item
  \textbf{Site topography} --- convert aerial photography into a terrain
  mesh for use in civil design
\item
  \textbf{Integration with other APIs} --- translate the output mesh
  into OBJ or STL via the \hyperref[model-derivative]{Model Derivative}
  API for further analysis in R
\end{itemize}

\section{Before You Start}\label{before-you-start}

\begin{tcolorbox}[enhanced jigsaw, left=2mm, bottomrule=.15mm, opacityback=0, opacitybacktitle=0.6, bottomtitle=1mm, breakable, toptitle=1mm, colbacktitle=quarto-callout-tip-color!10!white, toprule=.15mm, titlerule=0mm, colframe=quarto-callout-tip-color-frame, colback=white, arc=.35mm, coltitle=black, leftrule=.75mm, rightrule=.15mm, title=\textcolor{quarto-callout-tip-color}{\faLightbulb}\hspace{0.5em}{Tip}]

\textbf{Image guidelines for best results:}

\begin{itemize}
\tightlist
\item
  Minimum \textbf{20 images} per scene; 50--200 is typical for a small
  structure
\item
  \textbf{60--80 \% overlap} between adjacent images --- every part of
  the subject should appear in at least 3 photos
\item
  Consistent, diffuse lighting; avoid harsh shadows or direct sun
  reflections
\item
  Keep the camera moving around the object, not zooming in from a fixed
  point
\item
  GPS/EXIF tags are optional but improve georeferencing accuracy
\end{itemize}

\end{tcolorbox}

\chapter{ACC Data Management}\label{acc-data-management-1}

The \href{https://aps.autodesk.com/en/docs/data/v2/overview/}{Data
Management (BIM 360 / ACC)} API exposes a hierarchical project file
structure used by AutoDesk Construction Cloud (ACC) and BIM 360. Unlike
the Object Storage Service (OSS) covered in the
\hyperref[data-management]{Data Management} chapter --- which uses flat
buckets --- the ACC hierarchy organises files into \textbf{Hubs →
Projects → Folders → Items → Versions}.

\begin{verbatim}
Hub (BIM 360 account or personal hub)
 └── Project
      └── Folder
           └── Item (a file)
                └── Version (a specific revision)
\end{verbatim}

This structure enables per-project permissions, version history, and
folder-level organisation across large teams.

\section{Why It Matters}\label{why-it-matters}

The OSS bucket API is ideal for programmatic file processing pipelines
where you manage storage yourself. The ACC hierarchy is required when
you need to:

\begin{itemize}
\tightlist
\item
  Navigate files that live inside a BIM 360 or ACC project
\item
  Access a specific version of a model checked in by a team member
\item
  Respect folder-level access controls configured by a project admin
\end{itemize}

\chapter{Webhooks}\label{webhooks-1}

The
\href{https://aps.autodesk.com/en/docs/webhooks/v1/overview/}{Webhooks
API} lets APS push HTTP notifications to a URL you control whenever a
specified event occurs --- for example, when a file translation
completes, a new model version is published, or a construction issue
changes status.

Using webhooks eliminates the need to poll \texttt{checkFile()} or
\texttt{checkPdf()} in a loop. Instead, APS calls your server the moment
the event fires.

\section{Why It Matters}\label{why-it-matters-3}

Polling works fine for interactive sessions, but in production pipelines
it wastes API quota and adds latency. A webhook-driven pipeline looks
like:

\begin{verbatim}
Upload file → POST /webhooks (register callback)
            ↓ (APS fires event when translation done)
     Your server receives POST → trigger downstream R job
\end{verbatim}

This is especially useful for automated reporting pipelines where a new
DWG published to ACC should automatically trigger translation, metadata
extraction, and a Shiny dashboard refresh.

\chapter{Troubleshooting}\label{troubleshooting-1}

\section{Common Issues}\label{common-issues}

Here is a running list of common issues that users have run into:

\textbf{Problem:} Some of the functions seem to work, others don't.

\textbf{Solution:} Make sure to sign up for the correct APIs and
services through AutoDesk Platform Services (APS) at
\url{https://aps.autodesk.com}.

\textbf{Problem:} I can't create bucket ``mybucket''.

\textbf{Solution:} Try a unique bucket name.

\textbf{Problem:} I was able to create a bucket and upload a file, but I
can't translate the file.

\textbf{Solution:} Each task requires different tokens with different
scopes. Make sure to create a token with the
\texttt{data:read\ data:write} scopes in order to translate a file.
